0. 一句话总览

这一波 AI 工具的核心变化，不是“模型能聊天”，而是“模型能在受控环境里带着上下文使用工具完成任务”。

可以把整个体系理解成：

如果用软件工程语言类比：

  • 模型像推理引擎，不等于完整应用。
  • Agent像带执行循环的应用层，把模型、上下文和工具组织起来。
  • Tools像函数、命令、浏览器、数据库、检索器等可执行能力。
  • MCP像一种标准化外设协议，让不同工具和数据源以统一方式接入 AI 应用。
  • Skills像可复用的 SOP、插件说明书或任务手册，告诉 Agent 遇到某类任务该怎么做。
  • Codex是专注软件工程场景的编码 Agent，可以读仓库、改文件、运行命令、验证结果、写说明。

1. 学习路线图

建议按下面顺序学习：

  1. LLM 基础：模型如何根据上下文生成输出，为什么上下文质量决定结果上限。
  2. 工具调用：模型不只回答，还能选择调用函数、命令、检索、浏览器等工具。
  3. Agent Loop：计划、行动、观察、修正、交付的循环。
  4. 上下文工程：把目标、约束、历史、文件、工具结果组织成模型可用的信息。
  5. Codex：把 Agent Loop 应用到代码仓库和软件工程。
  6. Skills：把重复任务固化成可复用工作流。
  7. MCP：把外部工具、数据、提示模板标准化接入。
  8. 插件与扩展：把 Skills、MCP、命令、资源打包分发。
  9. 安全与治理：权限、沙箱、审批、日志、评测、数据边界。
  10. 实践项目：让 Agent 真正完成一个有验收标准的任务。

2. 大图：现代 AI Agent 系统由什么组成

一个现代 Agent 系统一般包含这些层：

层级 作用 常见例子
用户目标层 描述要完成什么 “修复这个 bug”“整理一份学习笔记”“分析这份表格”
指令层 约束做事方式 system/developer/user prompt、AGENTS.md、技能说明
上下文层 提供任务所需信息 文件内容、搜索结果、历史对话、代码结构、日志
推理层 模型判断下一步 计划、分解、选择工具、总结观察
工具层 执行外部动作 Shell、浏览器、文件编辑、数据库、API、检索
协议层 标准化能力接入 MCP、函数调用 schema、插件 manifest
记忆层 保存偏好或长期状态 用户偏好、项目约定、历史结论
安全层 限制与审计行为 沙箱、审批、权限、日志、速率限制
评测层 判断结果好不好 单元测试、E2E、人工验收、自动评分

关键点：
Agent 不是模型本身，而是一个围绕模型构建的执行系统。

3. Agent：从“回答问题”到“完成任务”

3.1 Agent 的定义

在实际工程里，Agent 通常指一个能够：

  • 理解目标；
  • 读取上下文；
  • 制定下一步；
  • 调用工具；
  • 观察工具结果；
  • 根据结果调整策略；
  • 直到交付或明确失败；

3.2 Agent 与 Chatbot 的区别

维度 Chatbot Agent
主要行为 生成回答 完成任务
外部动作 通常没有或很少 经常调用工具
上下文 对话文本为主 对话、文件、网页、日志、数据库、命令输出
结果 文本回复 文件、代码变更、报告、自动化动作、可验证状态
风险 错答 错答加错误行动
工程重点 Prompt 质量 工具边界、状态管理、权限、安全、评测

3.3 一个好的 Agent 需要什么

好的 Agent 不只是“更聪明的模型”，还需要：

  • 清晰目标：什么算完成，什么不算完成。
  • 可靠工具：工具接口稳定，错误可读，结果可检查。
  • 上下文压缩：不要把所有东西都塞给模型，要给相关信息。
  • 可恢复执行：工具失败、网络失败、测试失败时能换路线。
  • 安全边界：不能随意删除、上传、付费、发送敏感信息。
  • 可观测性：知道它调用了什么工具、输入输出是什么。
  • 评测机制：靠测试、检查、指标和人工验收判断质量。

3.4 常见 Agent 失败模式

  • 目标漂移：做着做着忘了原始目标。
  • 工具滥用：能搜索就一直搜索，能改文件就大范围重构。
  • 上下文污染：网页或文件里的恶意指令影响 Agent 行为。
  • 过早确信：没有验证就给结论。
  • 隐式副作用：不该提交表单、不该删除文件、不该发送数据却做了。
  • 不可复现：没有记录命令、数据来源和变更理由。

4. Codex：面向软件工程的 Agent

4.1 Codex 是什么

Codex 可以理解为 OpenAI 面向软件开发任务的编码 Agent。它的重点不是单次补全代码，而是围绕一个代码仓库执行软件工程任务，例如：

  • 阅读项目结构；
  • 理解现有模式；
  • 修改代码；
  • 运行测试；
  • 调试错误；
  • 做代码审查；
  • 生成文档；
  • 根据仓库约定持续工作。

在 Codex 这样的环境里，Agent 不只是“建议你怎么改”，而是可以实际操作工作区：

4.2 Codex 的核心价值

Codex 的价值在于把 AI 放进真实工程上下文：

  • 仓库上下文：读真实代码，而不是只靠用户粘贴片段。
  • 命令执行：可以运行测试、构建、lint、脚本。
  • 文件编辑：能直接修改项目文件。
  • 迭代验证：失败后能根据日志继续修复。
  • 工程约束：遵守项目约定、目录结构、测试策略。
  • 协作输出：告诉你改了什么、验证了什么、还有什么风险。

4.3 Codex 与普通 IDE AI 插件的区别

维度 普通 IDE 补全/聊天 Codex 式编码 Agent
工作单位 单个文件或选区 整个任务
上下文 编辑器附近内容 仓库、命令、日志、用户目标
行动能力 建议代码或插入代码 读写文件、运行命令、验证
交付方式 代码片段 可运行变更和结果说明
风险控制 编辑器权限为主 沙箱、审批、工作区边界、命令策略

4.4 Codex 里的几个重要概念

AGENTS.md

AGENTS.md 通常用于给 Agent 提供项目级长期约定，例如：

  • 项目如何安装依赖；
  • 如何运行测试；
  • 代码风格；
  • 哪些目录不能改；
  • PR 或 review 的格式；
  • 团队特定约定。

它像是给 AI 的“项目贡献指南”。
如果任务只在当前对话生效，用 prompt 就够；如果约定要长期跟随仓库，才适合放进 AGENTS.md。

Sandbox

沙箱用于限制 Agent 的文件、网络、命令权限。
它解决的问题是：Agent 需要行动能力，但行动能力不能无限大。

典型权限边界包括：

  • 哪些目录可读；
  • 哪些目录可写；
  • 是否允许联网；
  • 哪些命令需要审批；
  • 是否允许删除、移动、安装依赖。

Tools

Codex 的工具可以包括：

  • Shell 命令；
  • 文件编辑；
  • 搜索；
  • 浏览器；
  • 图片生成；
  • MCP 工具；
  • 自动化工具；
  • 其他插件能力。

工具是 Agent 的“手”。模型负责判断和组织，工具负责执行和观察现实。

Worktree / Workspace

Workspace 是 Agent 当前可以工作的文件范围。
真实工程里要特别注意：

  • 不要误改不相关文件；
  • 不要覆盖用户正在改的内容；
  • 不要无故重置 git 状态；
  • 尽量让变更可审查、可回滚、可验证。

5. Tools：Agent 的行动接口

5.1 工具调用是什么

工具调用就是让模型输出一个结构化请求，由宿主程序执行，再把结果返回给模型。

概念上类似：

宿主系统执行后返回：

5.2 常见工具类型

工具类型 用途 风险点
文件读取 理解项目、资料、配置 读取敏感文件
文件编辑 修改代码、写文档 覆盖用户变更、误改范围过大
Shell 测试、构建、脚本、数据处理 删除文件、安装依赖、执行危险命令
浏览器 查资料、测试网页、下载公开资源 网页提示注入、登录态、表单提交
搜索 获取最新资料 过时或不可靠来源
数据库/API 查询和写入业务数据 权限、隐私、误写入
代码解释器 数据分析、生成文件、绘图 环境差异、依赖问题

5.3 工具设计的基本原则

好的工具接口要：

  • 参数清晰；
  • 错误信息明确；
  • 返回结构化数据；
  • 默认只读；
  • 有权限边界；
  • 对危险操作要求确认；
  • 结果可验证；
  • 支持幂等或可恢复。

不好的工具接口通常表现为：

  • 参数含糊；
  • 返回大段不可解析文本；
  • 默认执行高风险操作；
  • 错误只返回“失败”；
  • 没有审计日志；
  • 权限过大。

6. Skills：把可复用经验变成 Agent 能执行的工作流

6.1 Skill 是什么

Skill 可以理解为给 Agent 的可复用工作说明。
它通常包含：

  • 什么时候使用这个技能；
  • 应该遵守哪些规则；
  • 具体工作流；
  • 需要读取哪些参考文件；
  • 可以使用哪些脚本或模板；
  • 输出格式和质量标准。

一个 Skill 往往有一个核心文件，比如 SKILL.md。
Agent 遇到匹配任务时先读 Skill，再按 Skill 的工作流执行。

6.2 Skill 与 Prompt 的区别

维度 Prompt Skill
生命周期 当前对话或当前任务 可复用、可安装、可维护
内容 一次性指令 工作流、规则、引用资料、脚本
适用范围 临时要求 某类任务
维护方式 直接写在对话里 文件化、版本化
典型例子 “这次用中文回答” “生成图片时先读提示规范，再调用图片工具”

6.3 Skill 适合沉淀什么

适合做成 Skill 的内容：

  • 重复出现的任务流程；
  • 需要固定工具顺序的任务；
  • 有行业知识或内部标准的任务；
  • 需要引用模板、脚本、样例的任务；
  • 容易出错、必须强约束的任务。

不适合做成 Skill 的内容：

  • 一次性偏好；
  • 当前任务临时约束；
  • 变化极快且没有维护机制的事实；
  • 只有一句话就能说明的简单要求。

6.5 Skill 的价值

Skill 解决的是“经验复用”问题：

  • 让 Agent 不必每次重新学习流程；
  • 把团队经验文件化；
  • 减少提示词反复粘贴；
  • 让复杂任务有标准执行路径；
  • 可以配合脚本、模板和资源；
  • 更容易做版本管理。

6.6 Skill 的风险

Skill 也可能带来问题：

  • 过度复杂，导致 Agent 读太多无关内容；
  • 规则冲突，和用户当前目标相矛盾；
  • 过时，没有维护；
  • 写得太抽象，无法执行；
  • 把本该是工具权限的问题写成自然语言约束，控制力不足。

写 Skill 时要遵守一个原则：
Skill 应该降低任务复杂度，而不是制造更多解释成本。

7. MCP：Model Context Protocol

7.1 MCP 是什么

MCP 是 Model Context Protocol，中文可理解为“模型上下文协议”。
它的目标是让 AI 应用用标准方式连接外部工具和数据源。

直观类比：

  • USB-C 统一了很多外设连接方式；
  • HTTP 统一了 Web 请求方式；
  • LSP 统一了编辑器和语言服务的交互；
  • MCP 试图统一 AI 应用和外部上下文/工具的交互方式。

MCP 官方文档将其定位为连接 AI 应用与外部工具、数据源、上下文的开放协议。

7.3 MCP 的基本角色

角色 作用
MCP Host 承载 AI 应用的宿主，例如桌面应用、IDE、Agent 平台
MCP Client Host 内部负责和 MCP Server 通信的组件
MCP Server 对外暴露工具、资源、提示模板等能力
Tool 可被模型调用的动作，例如查询 issue、读取文档、执行搜索
Resource 可读取的上下文数据，例如文件、文档、数据库记录
Prompt 可复用的提示模板或工作流入口

7.4 MCP 中 Tool、Resource、Prompt 的区别

类型 本质 是否有副作用 例子
Tool 动作 可能有 创建 issue、查询数据库、运行搜索
Resource 数据 通常只读 某个文档、文件、表结构
Prompt 模板 无直接副作用 “总结这个 PR 的提示模板”

理解这三者很重要。
不要把所有东西都做成 Tool。能只读的数据尽量做 Resource；可复用任务入口可以做 Prompt。

7.5 MCP 与 API 的区别

MCP 不是替代所有 API。
它更像是“给 AI 应用消费的 API 适配层”。

维度 普通 API MCP
面向对象 程序员/服务 AI Host / Agent
交互方式 自定义接口 标准协议
元数据 不一定为模型友好 通常描述工具、参数、资源
权限 由 API 自己管理 还要考虑 Host、Client、Server 权限边界
适合场景 业务服务调用 给 AI 应用接上下文和工具

7.6 MCP 的适用场景

适合用 MCP：

  • 多个 AI 客户端都要接同一个工具；
  • 工具有明确 schema；
  • 需要授权访问私有数据；
  • 希望工具描述、资源、提示模板标准化；
  • 希望团队维护一套共享 AI 工具服务。

不一定需要 MCP：

  • 只是一个项目内部的一次性脚本；
  • 工具只被一个程序调用且很简单；
  • 直接函数调用更清晰；
  • 需求尚未稳定。

7.7 MCP 的安全关注点

MCP 把外部能力接入 Agent，也带来了风险：

  • Server 是否可信；
  • Tool 是否有副作用；
  • 参数是否可能注入；
  • 返回内容是否可能提示注入；
  • 是否最小权限授权；
  • 是否记录调用日志；
  • 是否区分读写能力；
  • 是否需要人工审批。

最佳实践是：

  • 只安装可信 MCP Server；
  • 对写操作要求确认；
  • 把敏感数据最小化返回；
  • 给工具清晰命名和描述；
  • 避免把“删除、支付、发消息”等危险操作做成默认可调用能力；
  • 记录工具调用过程。

9. Context Engineering：比 Prompt Engineering 更大的问题

9.1 为什么上下文工程重要

Prompt Engineering 关注“怎么问”。
Context Engineering 关注“模型看到什么、按什么顺序看到、哪些信息被压缩、哪些信息被遗忘、哪些工具结果进入下一轮”。

对于 Agent，真正决定质量的往往不是一句神奇提示词，而是：

  • 给了哪些文件；
  • 文件是否相关；
  • 任务目标是否明确；
  • 工具结果是否结构化；
  • 是否保留关键约束；
  • 是否清理无关噪音；
  • 是否把临时结论和事实区分开。

9.2 上下文的类型

类型 内容 例子
指令上下文 必须遵守的规则 system/developer/user 指令、AGENTS.md
任务上下文 当前目标和验收标准 “生成详细学习笔记 Markdown”
环境上下文 当前运行环境 OS、cwd、日期、权限、工具
资料上下文 外部知识 官方文档、网页、论文
工作上下文 已经做过什么 命令输出、文件 diff、错误日志
记忆上下文 长期偏好 用户喜欢中文、项目命名习惯

9.3 上下文工程原则

  • 相关性优先：不要给模型一切，只给完成任务需要的信息。
  • 来源清楚：事实要知道来自哪里。
  • 层级清楚：系统规则、用户目标、网页内容不能混在一起。
  • 及时更新：工具结果改变后，旧判断要修正。
  • 压缩但不丢关键约束：长任务中要保留目标、限制、已验证事实。
  • 区分事实和推断：网页说了什么、你推断了什么，要分开。

10. RAG、Memory、Long Context 与 Agent 的关系

10.1 RAG

RAG 是 Retrieval-Augmented Generation，即检索增强生成。
它的核心是：

RAG 擅长回答知识库问题，但不等于 Agent。
Agent 可以使用 RAG 作为一个工具。

10.2 Memory

Memory 是长期记忆，用来保存用户偏好、历史选择、项目知识等。
注意：

  • Memory 不应该替代当前明确指令；
  • Memory 不应该保存敏感信息，除非用户明确授权；
  • Memory 适合偏好，不适合当作最新事实来源；
  • 当前任务的上下文通常比长期记忆更优先。

10.3 Long Context

长上下文模型可以一次读更多内容，但并不意味着可以无脑塞所有资料。
长上下文的问题包括：

  • 成本更高；
  • 注意力可能稀释；
  • 无关内容会干扰判断；
  • 仍然需要结构化和摘要。

长上下文解决“装得下”的问题；上下文工程解决“装什么、怎么装”的问题。

Tool Agent 可以执行的外部能力 应用层

12. Multi-Agent：多个 Agent 协作

Multi-Agent 指多个 Agent 分工协作。常见模式：

模式 说明
Manager-Worker 一个管理 Agent 拆任务，多个执行 Agent 完成子任务
Specialist 不同 Agent 专注不同领域，如代码、测试、文档、安全
Debate 多个 Agent 给出方案，相互质疑
Reviewer 一个 Agent 写，另一个 Agent 审查
Handoff 当前 Agent 判断自己不适合，交给另一个 Agent

Multi-Agent 的挑战：

  • 成本高；
  • 协调复杂；
  • 容易重复工作；
  • 结论冲突；
  • 需要共享状态和验收标准；
  • 需要防止错误在 Agent 间传播。

实际工程里，不要为了“多 Agent”而多 Agent。
只有当任务确实可拆分、角色边界清晰、并行收益明显时才值得使用。

13. Agent 安全：真正落地时最容易被低估

13.1 提示注入

提示注入是指外部内容试图改变 Agent 的指令。例如网页中写：

Agent 必须把网页、文档、邮件、搜索结果视为不可信内容。
它们可以提供事实，但不能覆盖用户和系统指令。

13.2 权限最小化

工具权限应遵循最小化原则：

  • 能只读就不要写；
  • 能限制目录就不要开放全盘；
  • 能限制域名就不要开放全网；
  • 能人工审批就不要默认执行高风险动作；
  • 能分离读写工具就不要混在一起。

13.3 高风险操作

高风险操作包括：

  • 删除或移动大量文件；
  • 提交、推送、发布；
  • 上传私有文件；
  • 发送邮件或消息；
  • 付款、下单、转账；
  • 修改权限或分享设置；
  • 安装不可信软件；
  • 写入生产数据库。

这些操作应该有明确授权、确认和日志。

13.4 可观测性

生产级 Agent 至少要记录：

  • 用户目标；
  • 关键上下文；
  • 工具调用；
  • 工具参数；
  • 工具结果；
  • 权限审批；
  • 最终输出；
  • 错误和恢复路径。

没有日志的 Agent 很难调试，也很难建立信任。

14. 评测：如何判断 Agent 做得好不好

Agent 的评测不能只看“回答像不像”。要看任务是否完成。

常见评测维度：

维度 问题
正确性 结果是否满足事实和逻辑
完成度 是否完成用户目标
工具使用 是否调用了合适工具
安全性 是否越权或泄露数据
稳定性 多次运行是否一致
成本 token、时间、工具调用是否合理
可解释性 是否能说明依据和过程
可恢复性 失败后是否能继续

软件工程场景尤其适合用自动测试评测：

  • 单元测试；
  • 集成测试；
  • E2E；
  • lint；
  • typecheck；
  • benchmark；
  • golden file；
  • regression test。

19. 常见误区

误区 1：Agent 越自主越好

不是。
生产系统里，自主性必须和权限、确认、审计一起设计。
越能行动的 Agent，越需要边界。

误区 2：有长上下文就不需要检索

不是。
长上下文解决容量，检索解决相关性。
真正高质量系统通常两者都用。

误区 3：MCP 等于插件市场

不是。
MCP 是协议，插件是打包和分发方式。
插件可以包含 MCP，但 MCP 本身不是插件市场。

误区 4：Skill 就是更长的 Prompt

不是。
Skill 更像工作流包，可以包含说明、脚本、模板、参考资料和质量标准。

误区 5：Agent 能运行测试就一定正确

不是。
测试只能覆盖已有断言。
Agent 还需要理解需求、检查边界、说明风险。

误区 6：工具越多越好

不是。
工具越多，选择空间越大，风险也越大。
好的 Agent 工具集应该小而清晰。

20. 术语表

术语 解释
LLM 大语言模型，负责生成、推理、理解上下文
Agent 能围绕目标循环调用工具并完成任务的系统
Agent Loop 计划、行动、观察、调整、交付的循环
Tool Agent 可调用的外部能力
Function Calling 模型输出结构化函数调用请求的机制
MCP Model Context Protocol，标准化连接外部工具和上下文的协议
MCP Server 暴露工具、资源、提示模板的服务
MCP Client AI Host 中连接 MCP Server 的组件
Resource MCP 中通常可读取的上下文数据
Prompt MCP 中可复用的提示模板
Skill 给 Agent 的可复用任务工作流说明
Plugin 可安装能力包，可能包含 Skills、MCP、脚本、资源
Codex 面向软件工程任务的编码 Agent
AGENTS.md 给 Agent 的仓库级长期指导文件
Sandbox 限制 Agent 文件、网络、命令权限的环境
RAG 检索增强生成，用外部资料补充上下文
Memory 长期保存的用户偏好或状态
Context Engineering 设计、选择、压缩和组织上下文的工程实践
Prompt Injection 外部内容试图覆盖系统或用户指令的攻击
Guardrail 约束 Agent 行为的规则、检查或安全机制
Evaluation 对 Agent 输出和行为进行测试、评分、验收

21. 推荐学习路径

第 1 周：建立概念地图

  • 学习 LLM、prompt、context、tool calling；
  • 手画 Agent Loop；
  • 区分 Agent、RAG、workflow automation；
  • 使用 Codex 完成几个小任务。

第 2 周：工具调用和 Codex 实践

  • 让 Codex 阅读一个小项目；
  • 让它修一个 bug；
  • 让它运行测试；
  • 让它写一份变更说明；
  • 观察它什么时候需要更多上下文。

第 3 周：Skills 和工作流沉淀

  • 写一个简单 Skill；
  • 为 Skill 增加 checklist；
  • 为 Skill 增加脚本；
  • 对比有 Skill 和无 Skill 的任务质量。

第 4 周：MCP 入门

  • 阅读 MCP 官方入门；
  • 理解 Host、Client、Server；
  • 区分 Tool、Resource、Prompt；
  • 运行一个最小 MCP Server；
  • 思考权限和安全边界。

第 5 周：工程化

  • 加日志；
  • 加审批；
  • 加错误恢复；
  • 加测试；
  • 加评测集；
  • 做一次完整复盘。
